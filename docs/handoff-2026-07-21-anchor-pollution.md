# Handoff — OK-anchor 오염 루프 검증과 존폐 결정 (2026-07-21)

이 문서는 anchor 작업의 **원인 문서**다: 어떤 질문에서 출발했고 무엇을
확인하기로 했는지를 기록한다. 짝을 이루는 결론/작업 문서는
[results-2026-07-21-anchor-pollution.md](results-2026-07-21-anchor-pollution.md).

## 출발점이 된 질문

TIER 0/1 오판정 논의([analysis-tier01-false-signals.md](analysis-tier01-false-signals.md)
FO-3 참조) 중 사용자 질문에서 출발했다:

> "사용자가 직접 OK하지 않는 한, 티어 0에서의 판정이 다시 판정 기준이 되는
> 일은 있나?"

확인 결과 **있다** — 그것이 anchor의 기본 설계다. anchor는 시스템 자신의
최근 eligible OK 판정 10건의 임베딩 평균이고(`sqlite.py:3373-3445`), 여기에
β=0.85를 곱한 값이 다음 페이지의 Tier 0 점수 후보로 들어간다
(`relevance.py`). 사용자 확인은 필요 없다. admission 가드
(`anchor_epsilon=0.05`)는 "anchor 단독 OK + exemplar 유사도 거의 0"인 극단
케이스만 차단하고, Tier 1 OK는 무조건 편입되는 우회로가 있었다
(`relevance.py:43`, 당시 기준).

## 1차 착수 결정 — 오염 루프 검증과 차단

이어진 사용자 제안: "티어 1급의 저렴한 모델에게 엄격한 검증을 시켜서,
유니크하면서 목적에 잘 부합하는 단어를 추가하게 한다거나. 벤치마크나 실험을
서브에이전트에게 돌려서 오염 루프가 어떻게 발생하는지 테스트해볼 수 있나?"

이에 따라 확인할 것을 정했다:

1. **재현**: 오염 루프("킬로그램" 실사고 유형)가 현행 스택(ONNX KoEn-E5-Tiny,
   τ=0.6, admission 가드, 접미사 스트리핑)에서 실제로 발생하는가.
2. **대책 비교**: ε 상향 / anchor 타이브레이커화 / 가중 anchor / LLM 검증
   승격(사용자 제안의 모사) 각각의 오염 차단력과 recall 부작용.
3. **반영**: 유효한 대책을 코드로 반영하고 테스트로 검증 (사용자 승인:
   "수정 2에 대한 요구를 넣어보고 테스트를 해서 결과를 한번 보자. 수정 1은
   필요한듯"). → PR #118.

실험 원칙: 실제 서버 함수와 실제 임베딩 모델을 사용하고, 리포 코드 무수정
상태에서 먼저 측정한 뒤 수정 전후를 비교한다.

## 2차 착수 결정 — anchor 존폐

1차 실험의 부수 발견("전 실험에서 anchor의 관련 페이지 구제 0건")이 후속
질문을 낳았다: "anchor의 현재 기여도를 평가해볼래? 원래 역할, 현재 수행하는
역할, Tier 0에 어떤 도움이 되는지."

평가 결과 원래 역할(정상 심화 credit, 지침서 §4.1)이 ONNX 전환의 τ 상향
(0.15→0.6)으로 구조적으로 불능이 된 정황이 확인되어, 대체·제거 실험을
추가로 진행하기로 했다:

4. **제거 실험(C1)**: anchor를 판정에서 완전히 빼면 무엇을 잃는가.
5. **대체 실험(C2/C3)**: Tier 1 OK를 세션 한정 exemplar로 승격하는 안 —
   무가드와 affinity 가드 변형의 오염 내성·이득 비교.
6. **최종 실험(C5)**: 안전판 3종(승격 가드 + LLM 키워드 검증 + 수명 제한 +
   promoted-only Tier 1 재확인)을 전부 갖춘 승격안의 순이득 측정.

결정 기준(사용자): **"C5가 C1 대비 순이득이 없으면 C1(제거) 채택."**
